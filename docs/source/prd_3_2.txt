1. 项目总览
1.1 项目定位
AutoPub 是面向海外市场的 AI 工具产品矩阵。第一阶段以 Chrome 插件矩阵为主，帮助用户完成网页图片
采集、AI 图片分析、视频创意拆解、SEO 页面诊断、竞品页面变化监控。第二阶段开发完整 SEO/GEO 
SaaS，承接插件用户和 Waitlist 用户，形成更高客单价的自动化增长平台。
1.2 当前阶段一句话定义
当前阶段的主交付物是 4 个可上线、可真实收款、可打包的 Chrome 插件；插件上线时必须打通 Creem 真
实支付闭环，用户可以完成付款，系统可以识别 Pro / Pro Plus / Credits 状态；SEO/GEO SaaS 当前只做
官网展示、Coming Soon、Notify Me 和 Waitlist 邮箱收集。

2. 产品战略
2.1 为什么先做插件矩阵

Chrome 插件比完整 SaaS 更轻，可以更快上线并获得真实用户反馈。

插件天然贴近浏览器工作流，用户看到问题时可以立刻使用。

多个插件对应多个获客入口：视觉素材、视频创意、SEO、竞品监控。

插件可以先验证 AI Credits 和 Pro 付费点，第一阶段必须通过 Creem 打通真实收款闭环，降低 SaaS 开发前
的不确定性。

插件用户和 Waitlist 邮箱可以沉淀到统一账号体系，为第二阶段 SaaS 导流。
2.2 为什么 SEO/GEO SaaS 放第二阶段
完整 SEO/GEO SaaS 涉及网站项目、整站抓取、SEO 规则、GEO 分析、内容生成、报告系统、订阅付费和
客户后台。如果第一阶段同时开发完整 SaaS，会显著增加周期、成本和失败风险。因此第一阶段只做展示
和收邮箱，第二阶段再完成 SaaS 主体。

插件可安装、可运行、可打包、可上架、可真实支付、
有测试报告。

12.4 Creem 支付技术要求

后端必须保存 creem_customer_id、creem_subscription_id、creem_checkout_id、
creem_product_id、creem_price_id。

Webhook 必须校验 Creem 签名，不能只信任前端跳转 success_url。

Webhook 事件必须幂等处理，同一个 creem_event_id 不允许重复发放 Credits。

支付成功后的权益更新以 webhook 为准，success_url 只负责用户体验展示。

插件端不得存储 Creem API Key；所有 checkout 创建和 webhook 处理必须在服务端完成。

插件端通过 /api/auth/me 或 /api/entitlements 获取当前 plan、credits_balance 和功能权限。

所有付费功能必须在后端二次校验 Credits / subscription，不允许只在前端隐藏按钮。
